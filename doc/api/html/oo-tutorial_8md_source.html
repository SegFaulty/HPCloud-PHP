1 Tutorial: Using HPCloud-PHP {#oo-tutorial}
2 =================
3 
4 HPCloud-PHP provides PHP language bindings for the HPCloud APIs. HPCloud
5 is an OpenStack-based cloud service offering a wide (and ever-expanding)
6 variety of services.
7 
8 In this tutorial, we will walk through the process of creating a simple
9 tool that interacts with HP Cloud's Object Storage. The emphasis in this
10 article is on getting started and learning the concepts, not building a
11 polished product.
12 
13 **This tutorial focuses on the object-oriented API.** The other way to
14 work with this library is through the stream wrapper. That topic is
15 covered in [another tutorial](@ref streams-tutorial).
16 
17 ## Pre-flight Check
18 
19 HPCloud-PHP has been developed to require PHP 5.3 or later. You are
20 strongly encouraged to also install the CURL PHP extension. Many
21 distributions of PHP come with this enabled. Sometimes, though, you may
22 need to do something like `apt-get php5-curl` or similar. (Don't take
23 our word for it -- check your system's documentation.)
24 
25 You can check for both of these conditions by checking the output of
26 `php --info` (on the commandline) or `<?php phpinfo(); ?>`.
27 
28 ### Check the pilot, too!
29 
30 In our pre-flight check, we would be remiss if we didn't point out that
31 there are some requirements for the pilot (that's you), too.
32 
33 The HPCloud library is composed of two parts:
34 
35 1. The Object-Oriented part, which is the subject of this tutorial.
36 2. The Stream Wrapper, which is the subject of another tutorial.
37 
38 The object-oriented library makes ample use of PHP namespaces. If you've
39 never seen these before, they look like this:
40 
41 ~~~{.php}
42 <?php
43 \HPCloud\Storage\ObjectStorage\RemoteObject
44 ?>
45 ~~~
46 
47 The namespace above is read like this: "The RemoteObject class is part
48 of the ObjectStorage package in the Storage package in the base HPCloud
49 package." Those familiar with Java, Python, and other languages will
50 recognize this way of talking (though the backslash is an unfortunate
51 symbol choice).
52 
53 For our library, we followed the recommendation of SPR-0, which means
54 that the class above can be found in the file at:
55 
56 ~~~
57 src/HPCloud/Storage/ObjectStorage/RemoteObject.php
58 ~~~
59 
60 The pattern of matching namespace to file name should (we hope) make it
61 easier for you to navigate our code.
62 
63 If this namespace stuff continues to confuse you, you may want to take a
64 look at [the PHP documentation](http://us3.php.net/manual/en/language.namespaces.php),
65 or you may just prefer to keep on reading and learn by example. We don't
66 do anything really fancy with namespaces.
67 
68 **In this document, we sometimes replace the backslash (\\) with double
69 colons (`::`) so that links are automatically generated.** So
70 `\HPCloud\Bootstrap` may appear as HPCloud::Bootstrap. The reason for
71 this is [explained elsewhere](@ref styleguide).
72 
73 ## Step 1: Getting the Library
74 
75 You can get the HPCloud-PHP library at the [HPCloud GitHub
76 Repository](https://github.com/hpcloud). The latest code is always
77 available there.
78 
79 The project also uses [Composer](http://packagist.org/), and this is the
80 best method for adding HPCloud-PHP to your PHP project.
81 
82 For our example, we will assume that the library is accessible in the
83 default include path, so the following line will include the
84 `Bootstrap.php` file:
85 
86 ~~~{.php}
87 include 'HPCloud/Bootstrap.php';
88 ~~~
89 
90 ## Step 2: Bootstrap the Library
91 
92 The first thing to do in your application is make sure the HPCloud
93 library is bootstrapped. When we say "bootstrap", what we really mean is
94 letting the library initialize itself.
95 
96 Bootstrapping does not always involve any manual interaction on your
97 part. If you are using an SPR-0 autoloader that knows of the HPCloud
98 directory, that is enough for the system to initialize itself.
99 
100 Sometimes, though, you will need to bootstrap HPCloud in your own code,
101 and this is done as follows:
102 
103 ~~~{.php}
104 <?php
105 require_once 'HPCloud/Bootstrap.php';
106 
107 use \HPCloud\Bootstrap;
108 use \HPCloud\Services\IdentityServices;
109 use \HPCloud\Storage\ObjectStorage;
110 use \HPCloud\Storage\ObjectStorage\Object;
111 
112 \HPCloud\Bootstrap::useAutoloader();
113 ?>
114 ~~~
115 
116 The first line should be self-explanatory: We require the main
117 `Bootstrap.php` file (which contains the `Bootstrap` class).
118 
119 After that, we declare a list of namespaced objects that we will use.
120 This way we can refer to them by their short name, rather than by their
121 fully qualified name.
122 
123 The last line initializes the built-in HPCloud autoloader. What does
124 this mean? It means that this is the only `require` or `include`
125 statement you need in your code. The library does the rest of the
126 including for you, on demand, in a performance-sensitive way.
127 
128 There are some other fancy things that HPCloud::Bootstrap can do for
129 you. Most notably, you can pass configuration parameters into it. But
130 for the time being, we are good to go.
131 
132 Our library is boostrapped. Next up: Let's connect to our account.
133 
134 ## Step 3: Connecting
135 
136 Our programming goal, in this tutorial, is to interact with the Object
137 Storage service on HP Cloud. (Object Storage is, for all intents and
138 purposes, basically a service for storing files in the cloud.)
139 
140 But before we can interact directly with Object Storage, we need to
141 authenticate to the system. And to do this, we need the following four
142 pieces of information:
143 
144 - Account ID: The account number.
145 - Secret key: The shared secret that, when paired with account number,
146  authenticates the client.
147 - Tenant ID: An identifier that maps an account to a set of services.
148  (In theory at least, one account can have multiple tenant IDs, and one
149  tenant ID can be linked to multiple accounts.)
150 - Endpoint URL: The URL to the Identity Services endpoint at HPCloud.
151 
152 Before you issue a forlorn sigh, envisioning some laborious task, let us
153 point out that all of this information is available in one place, Log
154 into [the console](https://console.hpcloud.com) and go to the `API Keys`
155 page. It's all there.
156 
157 ### Identity Services
158 
159 The HPCloud is composed of numerous services. There's the Compute
160 service, the Object Storage service, the CDN service... and so on.
161 
162 Authenticating separately to each of these would be a collosal waste of
163 network resources. And behind the scenes, account management would be
164 difficult on the server side.
165 
166 That's where Identity Services comes in. It is a central service that
167 handles all things authorization and authentication related. Roughly,
168 it works as follows:
169 
170 - The client sends an authentication request
171 - If it fails, the service returns an error
172 - If authentication succeeds, the service returns a time-sensitive token
173  (basically a shared secret) and a "service catalog"
174 
175 The *token* is valid for some fixed period of time (say, 30 minutes),
176 during which time it can be used for every other service. Each request
177 to an HPCloud service should send (along with other info) the token. The
178 remote service then validates the token with identity services, saving
179 our app the trouble of making another round trip.
180 
181 The *service catalog* lists all of the HPCloud services that the present
182 account can access.
183 
184 ### Authenticating
185 
186 With that little bit of theory behind us, we can now go about
187 authenticating.
188 
189 ~~~{.php}
190 <?php
191 $account = 'ADD ACCOUNT HERE';
192 $key = 'ADD KEY HERE';
193 $tenantId = 'ADD TENANT ID HERE';
194 $endpoint = 'ADD ENDPOINT URL HERE';
195 
196 $idService = new \HPCloud\Services\IdentityServices($endpoint);
197 $token = $idService->authenticateAsAccount($account, $key, $tenantId);
198 ?>
199 ~~~
200 
201 Assuming the variables above have been set to include valid data, this
202 script can connect to HPCloud and authenticate.
203 
204 When we construct a new HPCloud::Services::IdentityServices object, we must pass it the
205 endpoint URL for HPCloud Identity Services. Typically, that URL will
206 look something like this:
207 
208 ~~~
209 https://region-a.geo-1.identity.hpcloudsvc.com:35357
210 ~~~
211 
212 The `authenticateAsAccount()` method will authenticate to the
213 Identity Services endpoint. For convenience, it returns the
214 authorization token (`$token`), though we can also get the token from
215 `$idService->token()`.
216 
217 Note that the `IdentityServices` object may throw various exceptions
218 (all subclasses of HPCloud::Exception) during authentication. Failed
219 authentication results in an HPCloud::Transport::AuthorizationException, while
220 a network failure may result in an HPCloud::Transport::ServerException.
221 
222 Earlier, we talked about the service catalog. Once we've authenticated,
223 we can get the service catalog from `$idService->serviceCatalog()`. It
224 is an associative array, and you can get an idea of what it contains by
225 dumping it with `var_dump()`, should you so desire.
226 
227 At this point, we have what we need from Identity Services. It's time to
228 look at Object Storage.
229 
230 ### IdentityServices in a Nutshell
231 
232 Instances of HPCloud::Services::IdentityServices are responsible for:
233 
234 - Authentication
235 - Accessing the service catalog
236 - Accessing account info
237 - Associating tenant IDs with accounts (advanced)
238 
239 ## Step 4: Connecting to Object Storage
240 
241 The Object Storage system is concerned with two classes of things:
242 
243 - An Object: A self-contained bundle of data (back in my day, we called
244  them "files").
245 - A Container: A storage container (bucket) for objects.
246 
247 Your object storage can have any number of containers, and each
248 container can have any number of objects.
249 
250 In the object model for the HPCloud PHP library, a top-level object
251 called HPCloud::Storage::ObjectStorage provides access to the Object
252 Storage service. In this step, we will be working with that object.
253 
254 ### Getting an ObjectStorage Instance
255 
256 Earlier, we created an `IdentityServices` instance called `$idService`.
257 We will use that here to get the service catalog. Once we have the
258 catalog, we can have a new `ObjectStorage` instance created for us,
259 configured to talk to our account's Object Storage instance in the
260 HPCloud. Along with the service catalog, we also need our token that
261 shows the Object Storage endpoint that we have already authenticated to
262 Identity Services. Earlier, we captured that value in the `$token`
263 variable.
264 
265 Now we can get a new HPCloud::Storage::ObjectStorage instance:
266 
267 ~~~{.php}
268 <?php
269 $catalog = $idService->serviceCatalog();
270 
271 $store = ObjectStorage::newFromServiceCatalog($catalog, $token);
272 
273 // UPDATE: As of Beta 6, you can use newFromIdentity():
274 // $store = ObjectStorage::newFromIdentity($idService);
275 ?>
276 ~~~
277 
278 First we get the service catalog (`$catalog`), and then we use the
279 `ObjectStorage::newFromServiceCatalog()` static method to create the new
280 Object Storage instance.
281 
282 The pattern of using a constructor-like static function is used
283 throughout the HPCloud PHP library. Inspired by Objective-C constructors
284 and the Factory design pattern, it makes it possible for a single class
285 to have multiple constructors.
286 
287 In particular, many top-level classes provide a
288 `newFromServiceCatalog()` constructor function, since these classes know
289 how to construct instances from a service catalog, thus freeing the
290 developer up from knowing the details of a service catalog entry.
291 
292 Now we have an `ObjectStorage` instance that is already configured to
293 talk to our HPCloud object storage service. Next, we can create a
294 container.
295 
296 ### ObjectStorage in a Nutshell
297 
298 Instances of HPCloud::Storage::ObjectStorage are responsbile for:
299 
300 - Providing high-level information about the Object Storage service
301 - Creating, deleting, loading, and listing Containers
302 - Modifying Container ACLs
303 - Attaching a HPCloud::Storage::CDN service object to a Container (advanced)
304 
305 ## Step 5: Adding a Container
306 
307 Before we can start putting objects (files) into our Object Storage
308 service, we need a place to put them. An Object Storage service can hold
309 numerous containers (and each container can have different access
310 controls -- a topic we won't get into here).
311 
312 Containers are represented in the library by the
313 HPCloud::Storage::ObjectStorage::Container class. And creating a
314 container is done by a method on the `ObjectStorage` object that we
315 created above:
316 
317 ~~~{.php}
318 <?php
319 $store->createContainer('Example');
320 $container = $store->container('Example');
321 ?>
322 ~~~
323 
324 Recall that `$store` is the name of our `ObjectStorage` instance. In the
325 first of the two lines above, we create a new container named `Example`.
326 Then in the second line, we get that container.
327 
328 Why is this two steps? The answer is that the HPCloud PHP library mimics
329 the architecture of the underlying API. This is two operations (which
330 means it requires two network requests to the remote host), and so we
331 must perform two operations.
332 
333 The `createContainer()` call actually creates the new container on the
334 cloud's Object Storage. The second call connects to the remote object
335 storage, and gets the new container. The container that is returned will
336 have some additional information, such as the amount of space it takes
337 up on the remote storage, and the access control rules for that
338 container. All of this information will be available on the `$container`
339 instance.
340 
341 Our `$container` instance is an instance of
342 HPCloud::Storage::ObjectStorage::Container. This object can be used not
343 only to find out about a container, but also to get information about
344 the objects in that container.
345 
346 Now that we have a `Container`, we can add an object.
347 
348 ### The Container in a Nutshell
349 
350 (Yes, we realize the irony of that title.)
351 
352 A HPCloud::Storage::ObjectStorage::Container instance is responsible for the following:
353 
354 - Accessing information about the container
355 - Creating, saving, deleting, and listing objects in the container
356 - Providing path-like traversal of objects
357 - Copying objects across containers
358 - Interacting with CDN objects (advanced)
359 - Loading a lightweight representation of an object without fetching the
360  entire object (more on this later).
361 
362 Among the features of a Container, it can act like an `Iterator` and is
363 `Countable`. That means you can loop through a Container in a `foreach`
364 loop and also use `count($container)` to find out the number of objects
365 in a Container.
366 
367 ## Step 6: Storing an Object
368 
369 Now we are ready to create an object, and then store it in our
370 container.
371 
372 Before diving too deeply, it is important to point out a detail: When
373 working with a remote data storage service, we are typically working
374 with a local copy and a remote copy. If our code isn't constructed
375 correctly, it is possible for these two to get out of sync.
376 
377 Earlier, we created a container directly on the remote side, and then
378 fetched the container. As we create an object, we are going to do the
379 opposite: We will create a local object, and then save it to the remote
380 storage. Later, we will fetch the remote object.
381 
382 ~~~{.php}
383 <?php
384 $name = 'hello.txt';
385 $content = 'Hello World';
386 $mime = 'text/plain';
387 
388 $localObject = new Object($name, $content, $mime);
389 $container->save($localObject);
390 ?>
391 ~~~
392 
393 In the code above, we create `$localObject` with a `$name`, some
394 `$content`, and a `$mime` type. Strictly speaking, only `$name` is
395 required.
396 
397 The HPCloud::Storage::ObjectStorage::Object class is used primarily to
398 describe a locally created object. Once we have our new `Object`, we can
399 save it remotely using the `save()` method on our `$container` object.
400 This will push the object to the remote object storage service.
401 
402 While we can continue manipulating `$localObject`, we are working with
403 the local version, not the latest version of what's on the server. This
404 is fine if what we are doing is writing more data. However, when
405 examining the content of the object, remember that we are working with
406 the local copy, and its properties may differ from the remote copy's.
407 
408 ### What if I call save() twice with the same Object?
409 
410 Objects, like files on a file system, are referenced by name. Any time
411 you `save()` an object, it will be pushed to the remote object storage
412 server, which will happily replace the old content with your newly
413 submitted content.
414 
415 Next let's turn to loading objects from the remote object storage.
416 
417 ### The Object in a Nutshell
418 
419 The HPCloud::Storage::ObjectStorage::Object instances are used for:
420 
421 - Creating a local object to be stored remotely
422 
423 This class is also the base class for the `RemoteObject` class that we
424 will look at later.
425 
426 The API is generally constructed so that a developer needn't worry about
427 the differences between an `Object` and a `RemoteObject`. But in all but
428 the edgiest of edge cases, you would only create an instance of
429 `Object`, never of `RemoteObject`.
430 
431 ## Step 7: Loading an Object
432 
433 Containers not only provide the methods for saving objects, but also for
434 loading objects. Thus, we can fetch the object that we just created:
435 
436 ~~~{.php}
437 <?php
438 $object = $container->object('hello.txt');
439 
440 printf("Name: %s \n", $object->name());
441 printf("Size: %d \n", $object->contentLength());
442 printf("Type: %s \n", $object->contentType());
443 print $object->content() . PHP_EOL;
444 ?>
445 ~~~
446 
447 The `$object` variable now references an instance of a
448 HPCloud::Storage::ObjectStorage::RemoteObject that contains the entire
449 object. `RemoteObject` represents an object that was loaded from the
450 remote server. Along with providing the features of the `Object` class
451 we saw earlier, it also provides numerous optimizations for working over
452 the network.
453 
454 Now that we have the object, we print out several pieces of information
455 -- `name()`, `size()`, amd `type()`. Then, using `content()`, we fetch
456 the content of the object.
457 
458 ### Lazily Loading an Object
459 
460 The method we used above to fetch the object is perfect for our needs.
461 It pulls the entire object down in a single request. But imagine this
462 scenario: Our object storage has large media files, and we don't know
463 at loading time whether or not we need to access the body content, or
464 just the other data about the object.
465 
466 It would be a time-consuming task to download the entire body of a large
467 media file if we don't actually use the body. On the other hand, from an
468 API standpoint it is great to be able to pass around a single object,
469 and not require the application to know whether or not the body has been
470 retrieved.
471 
472 The `RemoteObject` solves this problem using a technique known as "lazy
473 loading". That is, it can pull some of the data right away, but defer
474 fetching the rest of the data until that data is actually needed.
475 
476 To fetch an object this way, we can just swap out one line in the
477 example above:
478 
479 ~~~{.php}
480 <?php
481 $object = $container->proxyObject('hello.txt');
482 
483 printf("Name: %s \n", $object->name());
484 printf("Size: %d \n", $object->contentLength());
485 printf("Type: %s \n", $object->contentType());
486 print $object->content() . PHP_EOL;
487 ?>
488 ~~~
489 
490 Instead of using `object()`, we now use `proxyObject()`. This method
491 immediately loads the core data about the remote object, but defers
492 fetching the content until the content is requested.
493 
494 In the example above, then, one network request is issued by
495 `proxyObject()`, but another is initiated when `$object->content()` is
496 called.
497 
498 ### The RemoteObject in a Nutshell
499 
500 Instances of a HPCloud::Storage::ObjectStorage::RemoteObject offer the following features:
501 
502 - Access to an object stored on the remote object storage
503 - A proxying mechanism for lazily loading objects
504 - Support for loading via CDN (advanced)
505 - A stream-based API for using stream and file-based PHP functions
506 - Automatic tempfile-based caching for large objects (using
507  `php://temp`).
508 
509 `RemoteObject` instances can be updated and then passed to
510 `Container::save()` to update the copy on the server, too.

512 ## Summary
513 
514 At this point we have created a very basic script that connects to
515 HPCloud and works with object storage. Clearly, this only scratches the
516 surface of what the HPCloud PHP library does. But hopefully this is
517 enough to get you started with the library.

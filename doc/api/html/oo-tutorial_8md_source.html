00001 Tutorial: Using HPCloud-PHP           {#oo-tutorial}
00002 =================
00003 
00004 HPCloud-PHP provides PHP language bindings for the HPCloud APIs. HPCloud
00005 is an OpenStack-based cloud service offering a wide (and ever-expanding)
00006 variety of services.
00007 
00008 In this tutorial, we will walk through the process of creating a simple
00009 tool that interacts with HP Cloud's Object Storage. The emphasis in this
00010 article is on getting started and learning the concepts, not building a
00011 polished product.
00012 
00013 **This tutorial focuses on the object-oriented API.** The other way to
00014 work with this library is through the stream wrapper. That topic is
00015 covered in [another tutorial](@ref streams-tutorial).
00016 
00017 ## Pre-flight Check
00018 
00019 HPCloud-PHP has been developed to require PHP 5.3 or later. You are
00020 strongly encouraged to also install the CURL PHP extension. Many
00021 distributions of PHP come with this enabled. Sometimes, though, you may
00022 need to do something like `apt-get php5-curl` or similar. (Don't take
00023 our word for it -- check your system's documentation.)
00024 
00025 You can check for both of these conditions by checking the output of
00026 `php --info` (on the commandline) or `<?php phpinfo(); ?>`.
00027 
00028 ### Check the pilot, too!
00029 
00030 In our pre-flight check, we would be remiss if we didn't point out that
00031 there are some requirements for the pilot (that's you), too.
00032 
00033 The HPCloud library is composed of two parts:
00034 
00035 1. The Object-Oriented part, which is the subject of this tutorial.
00036 2. The Stream Wrapper, which is the subject of another tutorial.
00037 
00038 The object-oriented library makes ample use of PHP namespaces. If you've
00039 never seen these before, they look like this:
00040 
00041 ~~~{.php}
00042 <?php
00043 \HPCloud\Storage\ObjectStorage\RemoteObject
00044 ?>
00045 ~~~
00046 
00047 The namespace above is read like this: "The RemoteObject class is part
00048 of the ObjectStorage package in the Storage package in the base HPCloud
00049 package." Those familiar with Java, Python, and other languages will
00050 recognize this way of talking (though the backslash is an unfortunate
00051 symbol choice).
00052 
00053 For our library, we followed the recommendation of SPR-0, which means
00054 that the class above can be found in the file at:
00055 
00056 ~~~
00057 src/HPCloud/Storage/ObjectStorage/RemoteObject.php
00058 ~~~
00059 
00060 The pattern of matching namespace to file name should (we hope) make it
00061 easier for you to navigate our code.
00062 
00063 If this namespace stuff continues to confuse you, you may want to take a
00064 look at [the PHP documentation](http://us3.php.net/manual/en/language.namespaces.php),
00065 or you may just prefer to keep on reading and learn by example. We don't
00066 do anything really fancy with namespaces.
00067 
00068 **In this document, we sometimes replace the backslash (\\) with double
00069 colons (`::`) so that links are automatically generated.** So
00070 `\HPCloud\Bootstrap` may appear as HPCloud::Bootstrap. The reason for
00071 this is [explained elsewhere](@ref styleguide).
00072 
00073 ## Step 1: Getting the Library
00074 
00075 You can get the HPCloud-PHP library at the [HPCloud GitHub
00076 Repository](https://github.com/hpcloud). The latest code is always
00077 available there.
00078 
00079 The project also uses [Composer](http://packagist.org/), and this is the
00080 best method for adding HPCloud-PHP to your PHP project.
00081 
00082 For our example, we will assume that the library is accessible in the
00083 default include path, so the following line will include the
00084 `Bootstrap.php` file:
00085 
00086 ~~~{.php}
00087 include 'HPCloud/Bootstrap.php';
00088 ~~~
00089 
00090 ## Step 2: Bootstrap the Library
00091 
00092 The first thing to do in your application is make sure the HPCloud
00093 library is bootstrapped. When we say "bootstrap", what we really mean is
00094 letting the library initialize itself.
00095 
00096 Bootstrapping does not always involve any manual interaction on your
00097 part. If you are using an SPR-0 autoloader that knows of the HPCloud
00098 directory, that is enough for the system to initialize itself.
00099 
00100 Sometimes, though, you will need to bootstrap HPCloud in your own code,
00101 and this is done as follows:
00102 
00103 ~~~{.php}
00104 <?php
00105 require_once 'HPCloud/Bootstrap.php';
00106 
00107 use \HPCloud\Bootstrap;
00108 use \HPCloud\Services\IdentityServices;
00109 use \HPCloud\Storage\ObjectStorage;
00110 use \HPCloud\Storage\ObjectStorage\Object;
00111 
00112 \HPCloud\Bootstrap::useAutoloader();
00113 ?>
00114 ~~~
00115 
00116 The first line should be self-explanatory: We require the main
00117 `Bootstrap.php` file (which contains the `Bootstrap` class).
00118 
00119 After that, we declare a list of namespaced objects that we will use.
00120 This way we can refer to them by their short name, rather than by their
00121 fully qualified name.
00122 
00123 The last line initializes the built-in HPCloud autoloader. What does
00124 this mean? It means that this is the only `require` or `include`
00125 statement you need in your code. The library does the rest of the
00126 including for you, on demand, in a performance-sensitive way.
00127 
00128 There are some other fancy things that HPCloud::Bootstrap can do for
00129 you. Most notably, you can pass configuration parameters into it. But
00130 for the time being, we are good to go.
00131 
00132 Our library is boostrapped. Next up: Let's connect to our account.
00133 
00134 ## Step 3: Connecting
00135 
00136 Our programming goal, in this tutorial, is to interact with the Object
00137 Storage service on HP Cloud. (Object Storage is, for all intents and
00138 purposes, basically a service for storing files in the cloud.)
00139 
00140 But before we can interact directly with Object Storage, we need to
00141 authenticate to the system. And to do this, we need the following four
00142 pieces of information:
00143 
00144 - Account ID: The account number.
00145 - Secret key: The shared secret that, when paired with account number,
00146   authenticates the client.
00147 - Tenant ID: An identifier that maps an account to a set of services.
00148   (In theory at least, one account can have multiple tenant IDs, and one 
00149   tenant ID can be linked to multiple accounts.)
00150 - Endpoint URL: The URL to the Identity Services endpoint at HPCloud.
00151 
00152 Before you issue a forlorn sigh, envisioning some laborious task, let us
00153 point out that all of this information is available in one place, Log
00154 into [the console](https://console.hpcloud.com) and go to the `API Keys`
00155 page. It's all there.
00156 
00157 ### Identity Services
00158 
00159 The HPCloud is composed of numerous services. There's the Compute
00160 service, the Object Storage service, the CDN service... and so on.
00161 
00162 Authenticating separately to each of these would be a collosal waste of
00163 network resources. And behind the scenes, account management would be
00164 difficult on the server side.
00165 
00166 That's where Identity Services comes in. It is a central service that
00167 handles all things authorization and authentication related. Roughly,
00168 it works as follows:
00169 
00170 - The client sends an authentication request
00171 - If it fails, the service returns an error
00172 - If authentication succeeds, the service returns a time-sensitive token
00173   (basically a shared secret) and a "service catalog"
00174 
00175 The *token* is valid for some fixed period of time (say, 30 minutes),
00176 during which time it can be used for every other service. Each request
00177 to an HPCloud service should send (along with other info) the token. The
00178 remote service then validates the token with identity services, saving
00179 our app the trouble of making another round trip.
00180 
00181 The *service catalog* lists all of the HPCloud services that the present
00182 account can access.
00183 
00184 ### Authenticating
00185 
00186 With that little bit of theory behind us, we can now go about
00187 authenticating.
00188 
00189 ~~~{.php}
00190 <?php
00191 $account = 'ADD ACCOUNT HERE';
00192 $key = 'ADD KEY HERE';
00193 $tenantId = 'ADD TENANT ID HERE';
00194 $endpoint = 'ADD ENDPOINT URL HERE';
00195 
00196 $idService = new \HPCloud\Services\IdentityServices($endpoint);
00197 $token = $idService->authenticateAsAccount($account, $key, $tenantId);
00198 ?>
00199 ~~~
00200 
00201 Assuming the variables above have been set to include valid data, this
00202 script can connect to HPCloud and authenticate.
00203 
00204 When we construct a new HPCloud::Services::IdentityServices object, we must pass it the
00205 endpoint URL for HPCloud Identity Services. Typically, that URL will
00206 look something like this:
00207 
00208 ~~~
00209 https://region-a.geo-1.identity.hpcloudsvc.com:35357
00210 ~~~
00211 
00212 The `authenticateAsAccount()` method will authenticate to the
00213 Identity Services endpoint. For convenience, it returns the
00214 authorization token (`$token`), though we can also get the token from
00215 `$idService->token()`.
00216 
00217 Note that the `IdentityServices` object may throw various exceptions
00218 (all subclasses of HPCloud::Exception) during authentication. Failed
00219 authentication results in an HPCloud::Transport::AuthorizationException, while
00220 a network failure may result in an HPCloud::Transport::ServerException.
00221 
00222 Earlier, we talked about the service catalog. Once we've authenticated,
00223 we can get the service catalog from `$idService->serviceCatalog()`. It
00224 is an associative array, and you can get an idea of what it contains by
00225 dumping it with `var_dump()`, should you so desire.
00226 
00227 At this point, we have what we need from Identity Services. It's time to
00228 look at Object Storage.
00229 
00230 ### IdentityServices in a Nutshell
00231 
00232 Instances of HPCloud::Services::IdentityServices are responsible for:
00233 
00234 - Authentication
00235 - Accessing the service catalog
00236 - Accessing account info
00237 - Associating tenant IDs with accounts (advanced)
00238 
00239 ## Step 4: Connecting to Object Storage
00240 
00241 The Object Storage system is concerned with two classes of things:
00242 
00243 - An Object: A self-contained bundle of data (back in my day, we called
00244   them "files").
00245 - A Container: A storage container (bucket) for objects.
00246 
00247 Your object storage can have any number of containers, and each
00248 container can have any number of objects.
00249 
00250 In the object model for the HPCloud PHP library, a top-level object
00251 called HPCloud::Storage::ObjectStorage provides access to the Object
00252 Storage service. In this step, we will be working with that object.
00253 
00254 ### Getting an ObjectStorage Instance
00255 
00256 Earlier, we created an `IdentityServices` instance called `$idService`.
00257 We will use that here to get the service catalog. Once we have the
00258 catalog, we can have a new `ObjectStorage` instance created for us,
00259 configured to talk to our account's Object Storage instance in the
00260 HPCloud. Along with the service catalog, we also need our token that
00261 shows the Object Storage endpoint that we have already authenticated to
00262 Identity Services. Earlier, we captured that value in the `$token`
00263 variable.
00264 
00265 Now we can get a new HPCloud::Storage::ObjectStorage instance:
00266 
00267 ~~~{.php}
00268 <?php
00269 $catalog = $idService->serviceCatalog();
00270 
00271 $store = ObjectStorage::newFromServiceCatalog($catalog, $token);
00272 ?>
00273 ~~~
00274 
00275 First we get the service catalog (`$catalog`), and then we use the
00276 `ObjectStorage::newFromServiceCatalog()` static method to create the new
00277 Object Storage instance.
00278 
00279 The pattern of using a constructor-like static function is used
00280 throughout the HPCloud PHP library. Inspired by Objective-C constructors 
00281 and the Factory design pattern, it makes it possible for a single class
00282 to have multiple constructors.
00283 
00284 In particular, many top-level classes provide a
00285 `newFromServiceCatalog()` constructor function, since these classes know
00286 how to construct instances from a service catalog, thus freeing the
00287 developer up from knowing the details of a service catalog entry.
00288 
00289 Now we have an `ObjectStorage` instance that is already configured to
00290 talk to our HPCloud object storage service. Next, we can create a
00291 container.
00292 
00293 ### ObjectStorage in a Nutshell
00294 
00295 Instances of HPCloud::Storage::ObjectStorage are responsbile for:
00296 
00297 - Providing high-level information about the Object Storage service
00298 - Creating, deleting, loading, and listing Containers
00299 - Modifying Container ACLs
00300 - Attaching a HPCloud::Storage::CDN service object to a Container (advanced)
00301 
00302 ## Step 5: Adding a Container
00303 
00304 Before we can start putting objects (files) into our Object Storage
00305 service, we need a place to put them. An Object Storage service can hold
00306 numerous containers (and each container can have different access
00307 controls -- a topic we won't get into here).
00308 
00309 Containers are represented in the library by the
00310 HPCloud::Storage::ObjectStorage::Container class. And creating a
00311 container is done by a method on the `ObjectStorage` object that we
00312 created above:
00313 
00314 ~~~{.php}
00315 <?php
00316 $store->createContainer('Example');
00317 $container = $store->container('Example');
00318 ?>
00319 ~~~
00320 
00321 Recall that `$store` is the name of our `ObjectStorage` instance. In the
00322 first of the two lines above, we create a new container named `Example`.
00323 Then in the second line, we get that container.
00324 
00325 Why is this two steps? The answer is that the HPCloud PHP library mimics
00326 the architecture of the underlying API. This is two operations (which
00327 means it requires two network requests to the remote host), and so we
00328 must perform two operations.
00329 
00330 The `createContainer()` call actually creates the new container on the
00331 cloud's Object Storage. The second call connects to the remote object
00332 storage, and gets the new container. The container that is returned will
00333 have some additional information, such as the amount of space it takes
00334 up on the remote storage, and the access control rules for that
00335 container. All of this information will be available on the `$container`
00336 instance.
00337 
00338 Our `$container` instance is an instance of
00339 HPCloud::Storage::ObjectStorage::Container. This object can be used not
00340 only to find out about a container, but also to get information about
00341 the objects in that container.
00342 
00343 Now that we have a `Container`, we can add an object.
00344 
00345 ### The Container in a Nutshell
00346 
00347 (Yes, we realize the irony of that title.)
00348 
00349 A HPCloud::Storage::ObjectStorage::Container instance is responsible for the following:
00350 
00351 - Accessing information about the container
00352 - Creating, saving, deleting, and listing objects in the container
00353 - Providing path-like traversal of objects
00354 - Copying objects across containers
00355 - Interacting with CDN objects (advanced)
00356 - Loading a lightweight representation of an object without fetching the
00357   entire object (more on this later).
00358 
00359 Among the features of a Container, it can act like an `Iterator` and is
00360 `Countable`. That means you can loop through a Container in a `foreach`
00361 loop and also use `count($container)` to find out the number of objects
00362 in a Container.
00363 
00364 ## Step 6: Storing an Object
00365 
00366 Now we are ready to create an object, and then store it in our
00367 container.
00368 
00369 Before diving too deeply, it is important to point out a detail: When
00370 working with a remote data storage service, we are typically working
00371 with a local copy and a remote copy. If our code isn't constructed
00372 correctly, it is possible for these two to get out of sync.
00373 
00374 Earlier, we created a container directly on the remote side, and then
00375 fetched the container. As we create an object, we are going to do the
00376 opposite: We will create a local object, and then save it to the remote
00377 storage. Later, we will fetch the remote object.
00378 
00379 ~~~{.php}
00380 <?php
00381 $name = 'hello.txt';
00382 $content = 'Hello World';
00383 $mime = 'text/plain';
00384 
00385 $localObject = new Object($name, $content, $mime);
00386 $container->save($localObject);
00387 ?>
00388 ~~~
00389 
00390 In the code above, we create `$localObject` with a `$name`, some
00391 `$content`, and a `$mime` type. Strictly speaking, only `$name` is
00392 required.
00393 
00394 The HPCloud::Storage::ObjectStorage::Object class is used primarily to
00395 describe a locally created object. Once we have our new `Object`, we can
00396 save it remotely using the `save()` method on our `$container` object.
00397 This will push the object to the remote object storage service.
00398 
00399 While we can continue manipulating `$localObject`, we are working with
00400 the local version, not the latest version of what's on the server. This
00401 is fine if what we are doing is writing more data. However, when
00402 examining the content of the object, remember that we are working with
00403 the local copy, and its properties may differ from the remote copy's.
00404 
00405 ### What if I call save() twice with the same Object?
00406 
00407 Objects, like files on a file system, are referenced by name. Any time
00408 you `save()` an object, it will be pushed to the remote object storage
00409 server, which will happily replace the old content with your newly
00410 submitted content.
00411 
00412 Next let's turn to loading objects from the remote object storage.
00413 
00414 ### The Object in a Nutshell
00415 
00416 The HPCloud::Storage::ObjectStorage::Object instances are used for:
00417 
00418 - Creating a local object to be stored remotely
00419 
00420 This class is also the base class for the `RemoteObject` class that we
00421 will look at later.
00422 
00423 The API is generally constructed so that a developer needn't worry about
00424 the differences between an `Object` and a `RemoteObject`. But in all but
00425 the edgiest of edge cases, you would only create an instance of
00426 `Object`, never of `RemoteObject`.
00427 
00428 ## Step 7: Loading an Object
00429 
00430 Containers not only provide the methods for saving objects, but also for
00431 loading objects. Thus, we can fetch the object that we just created:
00432 
00433 ~~~{.php}
00434 <?php
00435 $object = $container->object('hello.txt');
00436 
00437 printf("Name: %s \n", $object->name());
00438 printf("Size: %d \n", $object->contentLength());
00439 printf("Type: %s \n", $object->contentType());
00440 print $object->content() . PHP_EOL;
00441 ?>
00442 ~~~
00443 
00444 The `$object` variable now references an instance of a
00445 HPCloud::Storage::ObjectStorage::RemoteObject that contains the entire
00446 object. `RemoteObject` represents an object that was loaded from the
00447 remote server. Along with providing the features of the `Object` class
00448 we saw earlier, it also provides numerous optimizations for working over
00449 the network.
00450 
00451 Now that we have the object, we print out several pieces of information
00452 -- `name()`, `size()`, amd `type()`. Then, using `content()`, we fetch
00453 the content of the object.
00454 
00455 ### Lazily Loading an Object
00456 
00457 The method we used above to fetch the object is perfect for our needs.
00458 It pulls the entire object down in a single request. But imagine this
00459 scenario: Our object storage has large media files, and we don't know
00460 at loading time whether or not we need to access the body content, or
00461 just the other data about the object.
00462 
00463 It would be a time-consuming task to download the entire body of a large
00464 media file if we don't actually use the body. On the other hand, from an
00465 API standpoint it is great to be able to pass around a single object,
00466 and not require the application to know whether or not the body has been
00467 retrieved.
00468 
00469 The `RemoteObject` solves this problem using a technique known as "lazy
00470 loading". That is, it can pull some of the data right away, but defer
00471 fetching the rest of the data until that data is actually needed.
00472 
00473 To fetch an object this way, we can just swap out one line in the
00474 example above:
00475 
00476 ~~~{.php}
00477 <?php
00478 $object = $container->proxyObject('hello.txt');
00479 
00480 printf("Name: %s \n", $object->name());
00481 printf("Size: %d \n", $object->contentLength());
00482 printf("Type: %s \n", $object->contentType());
00483 print $object->content() . PHP_EOL;
00484 ?>
00485 ~~~
00486 
00487 Instead of using `object()`, we now use `proxyObject()`. This method
00488 immediately loads the core data about the remote object, but defers
00489 fetching the content until the content is requested.
00490 
00491 In the example above, then, one network request is issued by
00492 `proxyObject()`, but another is initiated when `$object->content()` is
00493 called.
00494 
00495 ### The RemoteObject in a Nutshell
00496 
00497 Instances of a HPCloud::Storage::ObjectStorage::RemoteObject offer the following features:
00498 
00499 - Access to an object stored on the remote object storage
00500 - A proxying mechanism for lazily loading objects
00501 - Support for loading via CDN (advanced)
00502 - A stream-based API for using stream and file-based PHP functions
00503 - Automatic tempfile-based caching for large objects (using
00504   `php://temp`).
00505 
00506 `RemoteObject` instances can be updated and then passed to
00507 `Container::save()` to update the copy on the server, too.

00509 ## Summary

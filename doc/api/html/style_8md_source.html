00016 ## Code Style
00017 
00018 The code in this package rigidly conforms to a given coding standard.
00019 The standard we use is published <a href="https://github.com/mattfarina/Coding-Standards">here</a> and is based
00020 on the Drupal coding standards, the PEAR coding standards, and several
00021 other popular standards.
00022 
00023 Important highlights:
00024 
00025 - Indentation uses *two space characters* and no tabs.
00026 - Variables and class names use CamelCasing (details above).

00028 Please do your best to follow coding standards when submitting patches.

00030 ### Object Orientation
00031 
00032 We have chosen to give the library a strongly object-oriented flavor.
00033 However, the stream wrapper integration provides a procudural interface
00034 to some of the services.
00035 
00036 ### Design Patterns and Coding Practices
00037 
00038 Where applicable, we use established coding patterns and practices. Some
00039 are PHP specific (like stream wrappers), while most enjoy broader
00040 industry support.
00041 
00042 There are a few things a developer should be aware of:
00043 
00044 **Accessors and Mutators**: The naming convention for methods on an
00045 object are as follows:
00046 
00047 - A function that accesses an object's data is a *noun*. Thus, the color
00048   of a fictional `Pillow` object may be accessed using
00049   `Pillow::color()`.
00050 - A function that performs an action is verbal, and this includes
00051   mutator functions (so-called "setters"). Thus,
00052   `Pillow::changeColor()`, `Pillow::setColor()`, and `Pillow::fight()` may
00053   all be appropriate mutator names. 
00054 - Unless a contract (interface or superclass) requires it, we do not ever
00055   define "getters".
00056 
00057 **Constructor Functions**: PHP does not support method overloading
00058 (that is, declaring multiple methods with the same name, but different
00059 signatures). While some languages (viz. Java, C++, C#) allow more than
00060 one constructor, PHP limits you to just one constructor.
00061 
00062 One strategy for working around this is to create constructors that take
00063 vague or generic parameters, and then perform various inspection tasks
00064 to figure out what the parameters are:
00065 
00066 ~~~{.php}
00067 <?php
00068 class Pillow {
00069 
00070   function __construct($name, $a2 = NULL, $a3 = NULL) {
00071 
00072     // ....
00073     if (is_string($a2)) {
00074       // Do one thing...
00075     }
00076     elseif (is_object($a2)) {
00077       // Do another thing.
00078     }
00079   }
00080 }
00081 ?>
00082 ~~~
00083 
00084 The above quickly degenerates into code that is both slow
00085 (because of the inspection tasks) and difficult to read and use.
00086 
00087 Another option, following Objective-C and Vala, is to create constructor
00088 functions. These are static functions (in PHP, at least) that can build
00089 instances. Constructor functions have signatures like
00090 `Pillow::newFromJSON()` and `Pillow::newFromXML()`.
00091 
00092 *This library uses constructor functions.* Generally, a very basic
00093 constructor is provided for cases where it is needed, but more complex
00094 cases are handled with specialized constructor functions.
00095 
00096 **Namespaces**: The library has been divided up into namespaces
00097 according to the following principles:
00098 
00099 - Each broad service category should have a namespace. Currently, the
00100   service categories are *Services* and *Storage*.
00101   * Services handle computing tasks on behalf of the client.
00102   * Storage handles data storage and retrieval
00103 - Individual services and storage services may have their own namespace
00104   if the number of supporting classes requires this.
00105 - The *Transport* namespace deals with lower-level details that are
00106   shared across all services.
00107 
00108 Otherwise, we have attempted to keep the namespace relatively shallow.
00109 
00110 ### Balancing Performance and Elegance
00111 
00112 Any network-based library must struggle with the inefficiencies of
00113 working over a network. This library is no exception. We've done our
00114 best to straddle the line between keeping load times down and making the
00115 code simple and elegant. Doubtless we have sometimes failed. Please feel
00116 free to suggest improvements on either side of the scales.
00117 
00118 ## Documentation Style
00119 
00120 This project is documented with Doxygen, and the configuration file used
00121 is available in ./config.doxy in this project.
00122 
00123 The following conventions are used:
00124 
00125 - In documentation, namespaces are separated with double-colon (::) instead of
00126   backslash characters (\\). This is because backslash characters have
00127   special meaning to Doxygen, and cannot be used as namespace separators.
00128 - We use \@retval instead of \@return. This special marker was added to
00129   Doxygen for languages like PHP where the return type is "optional". We
00130   try to always specify a return type, thus we use retval.
00131 
00132 ### Markdown and Doxygen >= 1.8
00133 
00134 Since Doxygen 1.8.0, Doxygen boasts native support for Markdown. Where
00135 possible, we write documentation in Markdown, which makes it both
00136 readable in the source code form, and also a fully integrated part of
00137 the API documentation.
